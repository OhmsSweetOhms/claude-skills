# RE102 Reference

RE102 is the radiated-emissions workflow. The local implementation currently
targets MIL-STD-461F-style scans with a Rigol RSA5065-TG and TBMA1B antenna
factor data.

## Measurement Meaning

An antenna-based RE102 result is field strength:

```text
dBuV/m = dBm + 106.9897 + antenna_factor_dB_per_m + cable_loss_dB - external_gain_dB
```

If `--equivalent-distance-m` is used, the project also applies:

```text
20*log10(measured_distance_m / equivalent_distance_m)
```

Use that only when intentionally normalizing between measurement distances.

For cabled signal-generator checks, bypass antenna factors. Those runs are
receiver input power:

```text
dBuV_50ohm = dBm + 106.9897
```

## Scan Planning

The current RE102 plan is bounded by antenna calibration breakpoints. For the
TBMA1B 1 m data, the full available range is 30 MHz to 3 GHz and currently
produces 83 scan segments. The analyzer only goes to 6 GHz, which is enough for
the current antenna data and project needs.

Preview without touching the analyzer:

```bash
.venv/bin/python tools/emi_control.py re102 plan --distance-m 1 --show 3
```

## UUT Campaign Pattern

Use `data/uuts/<uut_id>/<campaign_id>/re102/` for new UUT work. Keep ambient
and hot/config runs under `runs/`, with raw CSVs in `raw/`, corrected
`field_strength.json` in `derived/`, and SVGs in `plots/`. Do not put routine
quick-look plots under `reports/`; reserve `reports/` for signed-off report
packages.

For current vault work with the RSA, prefer the narrow-RBW engineering campaign
helpers. They produce a ready `run.json` first, then acquire only after the user
says "go".

Ambient setup:

```bash
.venv/bin/python tools/emi_control.py re102 init-engineering-run \
  --uut uut_001 \
  --date 2026-05-15 \
  --site vault \
  --role ambient \
  --polarization horizontal \
  --ping
```

Hot/config setup:

```bash
.venv/bin/python tools/emi_control.py re102 init-engineering-run \
  --uut uut_001 \
  --date 2026-05-15 \
  --site vault \
  --role hot_config \
  --config 1 \
  --polarization horizontal \
  --ping
```

Default vault profile:

- 30 MHz to 45 MHz at 300 Hz RBW/VBW.
- 45 MHz to 1730 MHz at 1 kHz RBW/VBW.
- 50 MHz segments after the leading 15 MHz segment.
- 10001 points, positive detector, max-hold trace, one sweep per segment.
- RSA only, preamp on, 0 dB input attenuation, -30 dBm reference level.
- Default cable loss is 2.4 dB for the PE300-72 plus PE300-36 path unless the
  bench setup changes. Keep the cable stack documented in JSON metadata; the
  reusable cable references live in the EMI repo `cables/` directory.

The generated `run.json` includes the `planned_command`. For "go", run that
command:

```bash
.venv/bin/python tools/emi_control.py re102 engineering-scan \
  data/uuts/uut_001/2026-05-15/re102/runs/002_hot_config_1_horizontal_vault_rsa/run.json
```

The RSA engineering scan uses CSV store plus FTP export. It waits for reported
sweep completion before export and rejects traces that still contain the
trailing `-170 dBm` floor from a partial sweep/clear. Preserve raw CSVs; reruns
or reprocessing should create new derived JSON/SVG artifacts rather than
editing raw files.

After acquisition, finalize:

```bash
.venv/bin/python tools/emi_control.py re102 finalize-run \
  data/uuts/uut_001/2026-05-15/re102/runs/002_hot_config_1_horizontal_vault_rsa/run.json
```

`finalize-run` marks the run acquired, recomputes peak and margin summary,
updates readiness flags, and writes the per-run SVG. Ambient-only data is a
baseline, not a standalone RE102 pass/fail claim.

## Plot And Report Pattern

Campaign plots should make ambient and UUT/config contributions separable:

- Individual run panels with the RE102 limit line.
- A combined summary panel for selected ambient/config traces.
- An ambient-delta panel where each config is plotted as config minus ambient.
- Peak or marker tables should describe worst margin or max over ambient, not
  hardcoded harmonic labels.
- Use enough logarithmic x-axis labels and visible tick marks for the 30 MHz to
  1730 MHz engineering range.

For generic comparisons, use `re102 compare-svg` or `re102 report-svg` with the
existing corrected JSONs. For the current UUT_001 SN010 campaign report, the
repo also has `emi/re102_uut001_sn010_report.py`; treat it as a report helper
owned by the repo, not as skill content to copy.

The older antenna-bounded `re102 scan` path still exists for formal or generic
30 MHz to 3 GHz scans. Use it only when the user explicitly wants that plan or
when the campaign does not call for the current narrow-RBW vault method.

## SMA100A System Checks

RE102 system checks use SMA100A CW tones plus narrow analyzer sweeps. Keep two
run modes separate because they answer different questions.

### Direct Analyzer Smoke Check

This is not formal RE102 calibration. It verifies the SMA100A, attenuation,
cabling, and analyzer narrow-sweep/export path in raw receiver power.

```text
SMA100A RF output -> required attenuation/cable -> analyzer input
```

Score this in `dBm` at the analyzer input:

```text
expected_analyzer_input_dbm =
    sma100a_power_dbm
  - source_to_analyzer_loss_db
  + source_to_analyzer_gain_db
```

Do not apply antenna factors, distance normalization, RE102 limits, or
`dBuV/m` reporting to this smoke check.

### Antenna-Port Measurement-System Check

This is the formal RE102 cabled measurement-system check. The SMA100A injects
a known CW level at the antenna connection point, and the analyzer performs a
narrow sweep over the injected tone.

```text
SMA100A RF output
  -> source attenuation/cable
  -> antenna-port injection plane
  -> measurement cable / gain path
  -> analyzer input
```

Existing single-tone repo pieces:

```text
emi_control.py re102 system-check-plan
emi_control.py re102 system-check-tone <frequency_hz>
emi_control.py re102 system-check-scan <frequency_hz>
emi_control.py re102 system-check-csv <rsa_trace.csv> <frequency_hz>
```

Point math:

```text
injected_dbuv_at_antenna_port =
    limit_dbuv_per_m
  - antenna_factor_db_per_m
  - margin_db

injected_dbm_at_antenna_port =
    injected_dbuv_at_antenna_port
  - 106.9897

sma100a_power_dbm =
    injected_dbm_at_antenna_port
  + injection_loss_db
  - injection_gain_db

expected_analyzer_input_dbm =
    injected_dbm_at_antenna_port
  - measurement_cable_loss_db
  + measurement_external_gain_db

expected_corrected_field_dbuv_per_m =
    limit_dbuv_per_m
  - margin_db
```

For TBMA1B calibration data, use interior calibration points rather than exact
antenna-factor endpoints:

```text
35 MHz, 100 MHz, 1 GHz, 2.9 GHz
```

Formal live automation is allowed only after these are documented:

- injection plane at the antenna connection point
- source-to-injection-plane cable and attenuator losses
- measurement-path cable loss and external gain into the analyzer
- selected frequency set
- source output limits and required external attenuation

Target multi-point command shape:

```bash
.venv/bin/python tools/emi_control.py re102 calibration-run re102_cal_YYYYMMDD \
  --siggen-host <sma100a_ip> \
  --host <rsa_ip> \
  --distance-m 1 \
  --limit aircraft_fixed_wing_internal_lt25m \
  --frequency-hz 35000000 \
  --frequency-hz 100000000 \
  --frequency-hz 1000000000 \
  --frequency-hz 2900000000 \
  --injection-loss-db <source_to_injection_loss_db> \
  --cable-loss-db <measurement_path_loss_db> \
  --apply
```

If `calibration-run` is not implemented in the active checkout, stage the
calibration folder and run the single-tone plan/tone/scan/csv commands one
point at a time. Save one combined `run.json` and `tone_plan.json`, include
per-point raw CSV and derived JSON paths, record amplitude error/pass-fail, and
turn SMA100A RF output off at the end including failure paths.

Preview the four-point plan without touching hardware:

```bash
.venv/bin/python tools/emi_control.py re102 system-check-plan \
  --distance-m 1 \
  --limit aircraft_fixed_wing_internal_lt25m \
  --frequency-hz 35000000 \
  --frequency-hz 100000000 \
  --frequency-hz 1000000000 \
  --frequency-hz 2900000000 \
  --injection-loss-db <source_to_injection_loss_db> \
  --cable-loss-db <measurement_path_loss_db>
```

## Known Bench Notes

- RSA IP observed in this project: `192.168.0.101`.
- RSA model observed: `RSA5065-TG`.
- Firmware observed: `00.01.03`.
- Working export path: SCPI stores a trace CSV, then FTP fetches it.
- Unsafe direct trace-data readback is fail-closed in the project driver; use
  CSV export unless a firmware-safe readback is proven.
- RM255 was an ambient lab scan with TBMA1B at 1 m and horizontal
  polarization. It is not compliance data because there was no screen room.
