2 · The three loops around one correlator PL.B3 · PS.B4–B9

In plain terms. There is one sensor — the correlator — and three steering wheels. The correlator slides a local replica of the satellite's code+carrier against the incoming signal and reports how well they line up (Early / Prompt / Late taps, each with I and Q). Each loop reads a different feature of that one report and turns a different knob: FLL reads how fast the prompt vector is spinning (frequency) → steers the carrier NCO coarsely; PLL reads its angle (phase) → steers the carrier NCO finely; DLL reads Early-vs-Late balance (code alignment) → steers the code NCO. The magic that makes it stable is the purple wire: the carrier loop's speed estimate is handed to the code loop (carrier aiding), so the code loop barely has to chase dynamics — it only cleans up the residual. Remove that wire and the DLL has to track the full Doppler on its own and falls apart. This is the same topology for both rails; only the block IDs, the rates, and the discriminator details differ.

Read it as three feedback rings sharing the correlator: FLL and PLL both steer the carrier NCO (green replica returns to de-rotate the input); DLL steers the code NCO (green replica shifts the E/P/L taps). The purple aiding wire is the only link between the carrier and code rings — it is what lets the DLL run at 0.5–2 Hz while the signal Dopplers by kHz. The gray dashed wires are not signal: they are the state machine reprogramming loop bandwidths and the E-L spacing as lock is gained.

Two conventions that have each cost a day (see gps-l1c.md §3, gps-tracking.md)
Carrier NCO is total-correction: freq = base + adj, never current + adj — the incremental form is a type-IV divergence (symptom: COARSE↔PULL_IN oscillation, PLI stuck ≈0.5).  ·  DLL sign is negated: code behind signal ⇒ L>E ⇒ discriminator<0 ⇒ the rate correction is subtracted to speed the code up.

3 · How the two rails are wired together C/A → L1C seeding · dual-rate · Δρ

In plain terms. L1C doesn't cold-start on its own here — C/A does the hard part first, then hands L1C a warm start. C/A's tracked Doppler tells L1C's FLL where to point (its pull-in window is only ±25 Hz); C/A's decoded time-of-week tells L1C which overlay chip and which 10 ms epoch it's looking at (L1C has no CNAV-2 decode yet). The two rails live on different clocks, so the C/A decimator's fixed delay must be booked back before their pseudoranges are compared.
LinkWhat crosses C/A → L1CWhy it exists / the failure it prevents
carrier seed C/A tracked Doppler (sub-Hz within ~2 s) seeds the L1C FLL — not an acquisition-grid value. L1C FLL pull-in ≈ ±25 Hz. The C/A acq grid (500 Hz) + L1C aided grid (50 Hz) can hand it 73 Hz of error → permanent COARSE/PULL_IN cycling.
overlay seed overlay_index = round(tow_ca(t₀)/10ms) mod 1800 The acquisition default (overlay hypothesis 0) is correct only at stream start; wrong everywhere else → corrupted pilot wipeoff.
time anchor integer epoch k = round((tow_ca − sub)/10ms); sub-epoch sub = (code_phase mod 10230)/1.023e6 stays L1C-measured. C/A supplies only the coarse integer 10 ms; the fine part is L1C's own code phase. Cloning the whole label makes Δρ ≡ 0 by construction (a real trap hit twice this thread).
dual-rate delay C/A t_rx label gets the PL.DECIMATOR group delay subtracted: 150 in-samples = 2195.7 m, a named constant in PS.B13. A wrong/missing compensation shows in Δρ as a fixed ~73 m·k ramp — that is the ADR-023 acceptance gate.
Δρ readout ρL1C − ρC/A at common 100 ms strobes, collected live during the run. Floor is ±7.32 m = ½ × 14.64 m (one 20.48 MSPS sample) — a label-quantization floor. Judge the mean; attribute the std, never tune it. Never strobe post-hoc: interp_tow_at rate-extrapolates from the newest two TOW labels, so querying after the run back-extrapolates every strobe from the end-of-run label pair — Δρ becomes a linear ramp vanishing at t_end (~0.7 ns of final-pair label jitter → ~220 m at a 22 s lever arm). This artifact was the entire "failing" F6 gate (see §4 note).

4 · The state machine as loop-parameter scheduler what each phase programs

In plain terms. The three loops don't use fixed settings — a supervisor widens everything to grab the signal, then tightens it for precision once locked. This table is the coupling between the loops and the lock detector: PS.B9's indicators decide the phase; the phase sets every loop's bandwidth and the correlator's E-L spacing. The two rails schedule differently: C/A uses the hard-switch heritage machine, L1C ports the FGI-GSRx 3-state machine (indicator-thresholded, no dwell timers) — which is why L1C passes through PULL_IN/COARSE in tens of milliseconds.

† The L1C per-state spacing schedule is a dead man walking — and the "failing" gate it was meant to fix was never a tracking problem

Resolution (2026-07-03, thread receiver/20260703-l1c-code-seed-rootcause): the closing Δρ gate that drove the whole spacing hunt was measuring its own evaluator, not the receiver. Instrumented replays show the tracker holding physical code error within ~±1 m for the entire 60 s run and the acquisition seeding within 0.040 chips worst-case (232/232 probes, zero side-peak captures) — while the post-hoc gate read +116/+15/+1.5/+67 m. The gate queried interp_tow_at after the run and silently back-extrapolated every strobe from the end-of-run label pair (§3, Δρ row). Strobed live, the same runs sit inside the ±7.32 m floor. The earlier seed-side-peak hypothesis (and fixes A/B/C: C/A code-phase seed · bump-jumping · BOC-robust discriminator) is refuted; the routed fix is a backward-extrapolation guard in PS.B13 plus a live-strobe gate evaluator (plan-02 of that thread).

The TMBOC spacing geometry itself is still real physics (real-Weil-code S-curves, receiver's own replica): the pilot ACF's main-peak capture region is ±0.336 chip (first zero ±0.352, vs C/A ±1.0), with stable false-lock equilibria near ±0.56 / ±0.71 chip (≈164 / 208 m). The normalized envelope discriminator slope is only weakly spacing-dependent where the geometry is sane (0.75× as half-spacing doubles 0.04→0.08) — but it collapses ~40× at d = 0.12 (an ACF shoulder: D′(0) ≈ −0.16) and inverts at d = 0.5 (D′(0) ≈ +2.09 — the origin is unstable and the DLL parks at ±0.269 chip). So the schedule above (0.5 → 0.12 → 0.04) is geometrically unsound in its first two states; it survives in practice only because the FGI state machine passes them in ~30 ms. The oracle (FGI-GSRx corrFingers,[-2 -0.04 0 0.04]) runs fixed ±0.04 in all states, and the acquisition's 0.040-chip worst-case seed gives that an 8.4× capture margin — the pending revert restores fixed 0.04. C/A needs none of this: its single-peaked triangle ACF makes 0.5 valid everywhere (open_sky pins pull_in = locked = 0.5; only urban narrows, 0.5→0.25).
